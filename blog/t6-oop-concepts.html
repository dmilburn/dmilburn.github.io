Composition vs. Inheritance

Quadrilaterals and Beethoven

2/20/15

One piece of sage advice that has been repeated a lot at DBC is "Don't Repeat Yourself(DRY)!" You can employ many different strategies to make your code more DRY. Two strategies that can help are employing inheritance or composition. While both of these strategies are about minimizing code, they have slightly different applications.

Inheritance

Inheritance has similarities with some basic math concepts.

An object with 4 sides is called a quadrilateral. Calling something a quadrilateral does not imply anything about the length of the sides or the angles between the sides, only that there are 4.

If we want specific info about length or angles, there are more specific names for different kinds of quadrilaterals such as: parallelograms (which have two sets of parallel sides) and trapezoids(which have one set of parallel sides). Since parallelograms and trapezoids both have 4 sides, they are both different kinds of quadrilaterals.

We are also taught about more specific kinds of parallelograms: rectangles (which have 4 90 degree angles) and rhombuses (which have 4 equal sides). Since rectangles and rhombuses have 2 sets of parallel sides, they are parallelograms. And since they both have 4 sides, they are also quadrilaterals.

In coding, inheritance can be thought of as "subclass IS A superclass". You can mark this in your code by writing in your class definition:

class Subclass < Superclass

If you were to create a class "Quadrilateral", you could assign it to have 4 sides. If you were to create a subclass of "Quadrilateral" called "Paralellogram", you could assign that to have 2 sets of parallel sides and it would automatically have 4 sides.

class Quadrilateral
 attr_reader :sides
 def sides
   @sides = 4
 end
end
class Parallelogram < Quadrilateral
 def initialize = 2
   sets_of_parallel_sides = 2
 end
end
If you were to create a new parallelogram and ask it how many sides it has:
parallelogram = Parallelogram.new 
puts parallelogram.sides 
it would print 4.

Composition

A composition is something that is greater than the sum of its parts.

Music is composed. A piece of music is really a collection of different notes. Yet, when all of these notes come together, they create something singular and beautiful.(Admittedly, I have trouble believing this with pointilism.)

In coding, composition can be thought of as "class HAS A different class". If you were to create a "Song" class, you would also need a "Note" class.

The first two bars of Beethoven's 5th symphony only has two different notes, G and Eflat.

If we were to try and recreate these bars using "Song" and "Note" classes, we would need these classes to be able to communicate. More specifically, "Song" needs to be able to get and use different instances of "Note". "attr_reader" is invaluable in this communication so that "Song" can invoke different methods inside of "Note". 

class Song
 def initialize
 end
 def add_pitch(*pitches)
   @array = pitches
 end
 def play
   puts @array.join("-") + "!"
 end
end
class Note
 attr_reader :pitch
 def initialize(pitch)
   @pitch = pitch
 end
 def pitch
   @pitch
 end
end

If we were to add the notes to the first two bars of Beethoven's fifth:

g = Note.new("G")

eflat = Note.new("EFlat")

beethoven_5 = Song.new

beethoven_5.add_pitch(g.pitch, g.pitch, g.pitch, eflat.pitch)

beethoven_5.play

it would result in: G-G-G-EFlat!

Summary

Think of inheritance as "subclass IS A superclass" and composition as "class HAS A different class". Use inheritance when you want to create something that is a subset of something you already created. Sandy Metz advises to only use inheritance when you feel pretty certain there won't be big changes. Use composition when you want a class to be able to have and use multiple instances of a different class.
